## 速答

当前测试覆盖**不能算完善**，但问题不在“后端完全没测”或“前端只能做 UI 测试”。证据显示，Rust 后端已经在 `commands`、`download`、`registry`、`models` 这些高风险纯逻辑边界上建立了不错的单测基础；真正薄弱的是 **前端没有正式测试入口**、**IPC 契约没有自动化对齐保护**、以及 **真实 Tauri 桌面链路还缺稳定 smoke**。

前端最值得先补的也不是视觉快照，而是运行时分支、状态聚合、事件驱动刷新和共享数据契约：`src/lib/tauri.ts` 的 `isTauriRuntime()` / `getLaunchReadiness()`，`App.tsx` 的非 Tauri 短路与自动更新 gating，`LaunchPanel.tsx` / `SettingsDialog.tsx` 的按钮 gating 与 focus trap，`UpdatePanel.tsx` 的错误映射、网络路由构造、安装事件刷新、恢复任务和安装历史展示，都是低成本高收益的补测边界。

现有门禁 `make check-lint` 更像“**Rust 测试 + TypeScript 类型检查 + 结构扫描**”，并不覆盖前端行为测试；而仓库虽然已经提供 `make tauri-dev-smoke` 作为真实桌面 smoke 入口，但它还不是一个被系统化验证的自动化回归层。因此，如果要做覆盖率完善，证据最支持的优先顺序是：**先补前端纯逻辑/组件测试与 IPC 契约测试，再补浏览器 smoke，最后把真实 Tauri 更新/安装/启动链路做成稳定 smoke**。

## 详细观察

### 1. 现有覆盖的真实含义

#### 后端：已经有一层不错的“纯逻辑防线”

本次直读到的 Rust 测试重点集中在四类边界：

1. **命令层输入规则与恢复入口**
   - `required_manifest_url`、`request_requires_manifest_url` 这样的规则已经被显式锁定。
   - 下载任务从 SQLite 恢复进 `DownloadManager` 的行为也有测试，不再只是内存态假设。

2. **下载/安装前的安全与恢复逻辑**
   - 下载文件的 `size` / `sha256` 校验有单测保护。
   - 恢复任务能区分 `verified` / `missing` / `corrupted` 缓存并决定 `can_install` / `can_retry`。
   - 缓存文件名生成也在防路径污染。

3. **注册表持久化**
   - 默认客户端、设置、下载任务、安装历史、启动探测写回都有测试样例。
   - 这意味着“安装历史 / 回滚路径 / 恢复任务”后端语义不是空白，而是已经进入可验证状态。

4. **跨端序列化契约**
   - `models.rs` 已验证 snake_case JSON 字段与状态值。
   - `src/types.ts` 与这些 shape 基本一一对应，说明前端完全可以围绕共享 fixture 做契约测试。

#### 前端：不是没测试点，而是还没接入测试体系

前端当前的主要问题不是“无法测试”，而是**仓库还没有把这些测试组织起来**：

- `package.json` 没有 `test` 脚本；
- `Makefile` 的 `make test` 只跑 Rust；
- `check-lint` 也不兜前端行为回归；
- 所以前端现在主要靠类型检查和人工 smoke，而不是自动化覆盖。

这也是为什么“前端测试不是 UI 之类的吧”这个判断是对的：从源码看，最容易先补、也最划算的，是下面这些逻辑边界，而不是上来就写重型视觉 E2E。

### 2. 前端最值得先补的测试边界

#### 2.1 单元测试优先对象

**优先级最高的一批纯逻辑函数：**

- `src/lib/tauri.ts`
  - `isTauriRuntime()`
  - `getLaunchReadiness()`
- `src/components/settings/SettingsDialog.tsx`
  - `routeHostFromUrl()`
  - `networkRouteUrl()`
  - `updateNetworkRoute()`
- `src/components/update/UpdatePanel.tsx`
  - `formatAssetSize()`
  - `formatCompletedAt()`
  - `progressPercent()`
  - `errorText()`
  - `buildNetworkRoute()`（可通过抽出 helper 或间接测试）

这批逻辑的共同特点是：

- 输入输出明确；
- 对 UI 样式不敏感；
- 一旦出错会直接影响运行时分支、错误提示或更新链路参数；
- 写测试的成本远低于写桌面端 E2E。

#### 2.2 组件测试优先对象

**最适合做交互级组件测试的，是“gating + 状态分支”而不是视觉布局：**

- `LaunchPanel.tsx`
  - 浏览器预览态下 `定位游戏` / `开始安装` / 主按钮是否正确禁用；
  - `ready` / `running` 状态时主按钮是否直接调用 `onPrimaryAction()`；
  - 其他状态时是否改为打开安装弹窗。

- `SettingsDialog.tsx`
  - `tauriRuntime=false` 时 browse / validate 是否禁用；
  - 打开弹窗时焦点是否落到第一个可聚焦元素；
  - `Escape` 是否关闭；
  - `Tab` / `Shift+Tab` 是否在对话框内循环。

- `UpdatePanel.tsx`
  - 非 Tauri 短路是否展示预览态提示；
  - 收到 `download-completed` / `install-completed` / `install-failed` 时是否刷新任务与历史；
  - 恢复任务和安装历史是否按后端返回数据渲染。

### 3. 契约测试应该怎么放

当前仓库已经天然具备做契约测试的三块拼图：

1. `src-tauri/src/main.rs` 的 20 个 command 注册面；
2. `src/lib/tauri.ts` 的集中封装；
3. `src/types.ts` 与 `src-tauri/src/test/models.rs` 的共享数据 shape。

因此，最有价值的契约测试不是“再读一遍代码”，而是：

- 用前端 fixture 固定 `LaunchReadiness`、`DownloadJob`、`DownloadJobRecovery`、`InstallHistoryRecord` 等形状；
- 用 Rust 现有序列化测试作为后端 truth source；
- 确保前端消费这些对象时不会因为字段名、状态枚举或 nullability 漂移而静默回归。

### 4. 浏览器 smoke 与真实 Tauri smoke 的分工

#### 浏览器 smoke

浏览器 smoke 主要保护的是：

- `App.tsx` / `UpdatePanel.tsx` 的非 Tauri 短路；
- `LaunchPanel.tsx` / `SettingsDialog.tsx` 的禁用态 UI；
- 页面在预览环境下不会因误触发 Tauri-only API 而崩溃。

也就是说，它更适合验证“**预览态安全降级是否仍然成立**”。

#### 真实 Tauri 桌面 smoke

真实桌面 smoke 才能覆盖：

- IPC 注入是否可用；
- `getDefaultClient` / `loadAppSettings` 的真实桥接；
- 下载事件与安装事件是否能从后端正确送达前端；
- 安装完成后客户端版本、恢复任务、安装历史是否真实刷新；
- 启动检测、运行中拦截、安装前关闭客户端等桌面约束。

仓库已经给了入口：`make tauri-dev-smoke`。但从当前结构看，它还更像“手动 smoke 入口”，还不是一套稳定、可回归的自动化验证层。这个缺口与 `#5 增加 Tauri 更新 smoke 验证` 任务是同一条主线。

### 5. 覆盖率完善的优先级建议

如果下一步要进入补测实现，当前证据最支持下面这个顺序：

#### P0：前端纯逻辑单元测试

先补：

- `isTauriRuntime()`
- `getLaunchReadiness()`
- `routeHostFromUrl()` / `updateNetworkRoute()`
- `formatAssetSize()` / `progressPercent()` / `errorText()`

原因：改动面小、收益高、能立刻把最容易回归的逻辑从“人工 eyeballing”转成“自动保护”。

#### P1：前端组件测试

先补：

- `LaunchPanel` 的 disabled/gating 与主按钮行为
- `SettingsDialog` 的 focus trap / ESC / Tab
- `UpdatePanel` 的非 Tauri 分支、事件刷新、恢复任务 / 安装历史渲染

原因：这些测试能直接保护“前端不是纯 UI，而是产品行为层”的核心分支。

#### P2：IPC 契约测试

围绕 `src/types.ts` 与 Rust `models` 序列化样例，固定几个关键 fixture：

- `LaunchReadiness`
- `AppSettings`
- `DownloadJob`
- `DownloadJobRecovery`
- `InstallHistoryRecord`

原因：跨端字段漂移是 Tauri 项目最容易出现、也最不容易靠类型检查单边发现的问题。

#### P3：浏览器 smoke

把当前浏览器预览态保护（非 Tauri 短路、按钮禁用、页面不崩）变成稳定的最小回归集。

#### P4：真实 Tauri 桌面 smoke

围绕默认客户端、更新检查、下载、安装、安装历史刷新、运行中拦截做最小闭环验证。这个优先级排在最后，不是因为它不重要，而是因为它最重、最脆弱，也最依赖前面三层先把基础逻辑收紧。

### 6. 与旧探索文档相比，需要修正的认知

当前这份测试覆盖探索也顺手纠正了几个会误导补测方向的旧结论：

- `main.rs` 当前注册面是 **20 个命令**，不是 19 个；
- `UpdatePanel.tsx` 已经监听 `install-progress` / `install-completed` / `install-failed`；
- 前端已经消费 `listInstallHistory()`，并且已经渲染“安装历史”区块。

因此，更新页的主要缺口不是“功能没接上”，而是“这些已接上的链路没有自动化保护”。这会直接影响补测优先级判断。
